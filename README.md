# FastAPI Mini Tutorial

This README introduces the core ideas behind a small FastAPI application:

- create an app
- run the server
- define routes
- read data from incoming requests
- return data to the client

FastAPI works especially well with Python type hints and Pydantic models. In many cases, you can describe your data with types, and FastAPI will handle parsing, validation, serialization, and API documentation for you.

## 1. Create and run a FastAPI app

Create a virtual environment, install FastAPI, and start with a file named `main.py`.

```bash
python -m venv .venv
source .venv/bin/activate
pip install "fastapi[standard]"
```

Create `main.py`:

```python
from fastapi import FastAPI

app = FastAPI()


@app.get("/")
async def root():
    return {"message": "Hello from FastAPI"}
```

Run the app:

```bash
fastapi dev main.py
```

You can then open:

- `http://127.0.0.1:8000/`
- `http://127.0.0.1:8000/docs`

The `/docs` page is automatically generated by FastAPI from your code.

## 2. Define routes

A route connects:

- an HTTP method such as `GET` or `POST`
- a URL path such as `/users` or `/users/{user_id}`
- a Python function that handles the request

Example:

```python
from fastapi import FastAPI

app = FastAPI()


@app.get("/hello")
async def say_hello():
    return {"message": "Hello"}


@app.post("/items")
async def create_item():
    return {"ok": True}
```

Common decorators are:

- `@app.get(...)`
- `@app.post(...)`
- `@app.put(...)`
- `@app.patch(...)`
- `@app.delete(...)`

## 3. How FastAPI gives data to your route handler

FastAPI looks at the parameters of your function and decides where each value comes from.

The five most common ways are:

1. in the URL path
2. in the query string
3. in the request body
4. in the request headers
5. in cookies

## 4. Path parameters

Path parameters are part of the URL itself.

Example URL:

```text
/people/basile/42
```

Example code:

```python
from fastapi import FastAPI

app = FastAPI()


@app.get("/people/{name}/{age}")
async def read_person(name: str, age: int):
    return {"name": name, "age": age}
```

If the client requests `/people/basile/42`, FastAPI passes:

- `name="basile"`
- `age=42`

Notice that `age` is declared as `int`. FastAPI converts it and validates it automatically.

## 5. Query parameters

Query parameters come after the `?` in the URL.

Example URL:

```text
/people?name=basile&age=42
```

Example code:

```python
from fastapi import FastAPI

app = FastAPI()


@app.get("/people")
async def read_person(name: str, age: int):
    return {"name": name, "age": age}
```

If a function parameter is not in the path and is a simple type like `str`, `int`, `float`, or `bool`, FastAPI treats it as a query parameter.

You can also make query parameters optional:

```python
@app.get("/search")
async def search(name: str | None = None, age: int | None = None):
    return {"name": name, "age": age}
```

## 6. Request body

For `POST`, `PUT`, and `PATCH` requests, data often comes from the request body as JSON.

The usual FastAPI pattern is to describe the body with a Pydantic model.

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()


class PersonIn(BaseModel):
    name: str
    age: int


@app.post("/people")
async def create_person(person: PersonIn):
    return person
```

If the client sends:

```json
{
  "name": "basile",
  "age": 42
}
```

FastAPI will:

- read the JSON body
- validate it
- create a `PersonIn` object
- pass it to the `person` parameter

## 7. Request headers

To read headers, use `Header`.

```python
from typing import Annotated

from fastapi import FastAPI, Header

app = FastAPI()


@app.get("/whoami")
async def whoami(
    user_agent: Annotated[str | None, Header()] = None,
):
    return {"user_agent": user_agent}
```

If the client sends the `User-Agent` header, FastAPI injects it into `user_agent`.

FastAPI automatically maps Python-style names such as `user_agent` to HTTP header names such as `User-Agent`.

## 8. Cookies

To read cookies, use `Cookie`.

```python
from typing import Annotated

from fastapi import Cookie, FastAPI

app = FastAPI()


@app.get("/session")
async def read_session(
    session_id: Annotated[str | None, Cookie()] = None,
):
    return {"session_id": session_id}
```

If the request contains a cookie named `session_id`, FastAPI injects it into the function parameter.

## 9. Returning data

FastAPI can automatically serialize common Python values and Pydantic models.

You can return:

- dictionaries
- lists
- strings
- numbers
- booleans
- Pydantic models

Examples:

```python
@app.get("/a")
async def route_a():
    return {"message": "hello"}


@app.get("/b")
async def route_b():
    return [1, 2, 3]
```

Returning a Pydantic model:

```python
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()


class PersonOut(BaseModel):
    name: str
    age: int


@app.get("/example")
async def example():
    return PersonOut(name="basile", age=42)
```

A simple pattern to follow is:

- use typed parameters for incoming data
- use Pydantic models for structured request bodies
- return dictionaries or Pydantic models

## 10. One example using all five input styles

```python
from typing import Annotated

from fastapi import Cookie, FastAPI, Header
from pydantic import BaseModel

app = FastAPI()


class MessageIn(BaseModel):
    text: str


class MessageOut(BaseModel):
    name: str
    age: int
    limit: int
    text: str
    user_agent: str | None
    session_id: str | None


@app.post("/messages/{name}/{age}")
async def create_message(
    name: str,
    age: int,
    limit: int,
    message: MessageIn,
    user_agent: Annotated[str | None, Header()] = None,
    session_id: Annotated[str | None, Cookie()] = None,
):
    return MessageOut(
        name=name,
        age=age,
        limit=limit,
        text=message.text,
        user_agent=user_agent,
        session_id=session_id,
    )
```

This route gets data from:

- the URL path: `name`, `age`
- the query string: `limit`
- the request body: `message`
- the request headers: `user_agent`
- the cookies: `session_id`

## 11. A useful rule of thumb

When you read a FastAPI route handler, you can usually guess where the data comes from with this mental model:

- if the parameter name appears in the path, it comes from the path
- if the parameter uses `Header(...)`, it comes from a header
- if the parameter uses `Cookie(...)`, it comes from a cookie
- if the parameter type is a Pydantic model, it usually comes from the request body
- if it is a simple typed parameter and not in the path, it usually comes from the query string

```python
from fastapi import Request


@app.get("/chat")
async def chat(request: Request):
    ...
```

Important details:

- the `request` parameter is required by FastAPI's template system
- even if your HTML file is static, you still pass the `request`

#### Return a template response

Inside the route, return `TemplateResponse(...)`:

```python
return templates.TemplateResponse(

    name="chat_0.html",
```

Important details:

- `name="chat_0.html"` means FastAPI should render `templates/chat_0.html`
- `context={}` means "there are no template variables to pass for now"

### SQLModel mini tutorial

For this exercise, we will use SQLModel.

SQLModel is a Python library that works well with FastAPI. It lets you define your data with classes, and then use those classes both as Python objects and as database tables.

You can think of `SQLModel` as something very close to Pydantic's `BaseModel`, but with extra database features added on top.

For now, you only need a few ideas.

#### A model class

With SQLModel, a class can represent a table:

```python
from sqlmodel import Field, SQLModel


class ChatMessage(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str
    message: str
```

Important details:

- `SQLModel` is the base class
- `table=True` means this class should become a database table
- `id` is the primary key
- `Field(default=None, primary_key=True)` means SQLite will generate the `id` automatically
- `id: int | None` means:
  - before the object is inserted into the database, the `id` can be `None`
  - after the database saves it, the `id` becomes an `int`

#### Create the database connection

To use SQLite, create an engine:

```python
from sqlmodel import create_engine

sqlite_url = "sqlite:///store.db"
engine = create_engine(
    sqlite_url,
    connect_args={"check_same_thread": False},
)
```

Important details:

- `store.db` is the SQLite database file
- to reset the database completely, it is okay to simply delete the `store.db` file
- `create_engine(...)` prepares the connection to the database
- `check_same_thread=False` is the usual setting used with SQLite in small FastAPI apps

#### Create the tables

Once the engine exists, create the tables:

```python
from sqlmodel import SQLModel


def create_db_and_tables():
    SQLModel.metadata.create_all(engine)
```

If the database file or table does not exist yet, SQLModel will create them.

#### Save one object

To write data into the database, open a session:

```python
from sqlmodel import Session


with Session(engine) as session:
    session.add(message)
    session.commit()
```

Useful ideas:

- `Session(engine)` opens a connection to work with the database
- `session.add(...)` prepares an object to be inserted
- `session.commit()` saves the change permanently

#### Read objects

To read rows from the database, use `select(...)`:

```python
from sqlmodel import Session, select


with Session(engine) as session:
    statement = select(ChatMessage).order_by(ChatMessage.id)
    messages = session.exec(statement).all()
```

Useful ideas:

- `select(ChatMessage)` means "read rows from the `ChatMessage` table"
- `order_by(ChatMessage.id)` keeps the messages in insertion order
- `session.exec(statement).all()` returns all matching rows

### Mini tutorial: SQLModel relationships

In exercise 2, the database only stored messages.

Now we want to store users too, and connect messages to users.

This is what relationships are for.

#### One-to-many

One-to-many means:

- one object on the left side can be linked to many objects on the right side

Example:

- one user can have many messages
- one user can have many sessions

#### Many-to-one

Many-to-one is the same relationship seen from the other side.

Example:

- many messages can belong to one user
- many sessions can belong to one user

#### Many-to-many

Many-to-many means:

- many objects on the left side can be linked to many objects on the right side

We are not using many-to-many yet in this exercise.

#### How SQLModel represents a relationship

Usually, you need two things:

1. a foreign key field
2. a `Relationship(...)`

Example:

```python
class User(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    name: str
    messages: list["ChatMessage"] = Relationship(back_populates="user")


class ChatMessage(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    message: str
    user_id: int = Field(foreign_key="user.id")
    user: User | None = Relationship(back_populates="messages")
```

Important details:

- `user_id` is the foreign key
- `foreign_key="user.id"` means "this field points to the `id` field of the `user` table"
- `Relationship(back_populates="...")` creates the Python relationship on both sides

For this exercise:

- one `User` has many `ChatMessage`
- one `ChatMessage` belongs to one `User`
- one `User` has many `UserSession`
- one `UserSession` belongs to one `User`

### Mini tutorial: password hashing

We should not store passwords in plain text.

If the database is leaked and passwords are stored in plain text, anyone reading the database can see them immediately.

So in this exercise, we will store a hash of the password instead of the password itself.

A hash is a transformed version of the password.

For this exercise, a simple version is enough:

```python
import hashlib


def hash_password(password: str) -> str:
    return hashlib.sha256(password.encode("utf-8")).hexdigest()
```

Why this is better than plain text:

- the real password is not stored directly in the database

Why this is still not enough for real security:

- `sha256` is too fast for password storage
- there is no salt in this exercise
- real applications should use stronger password hashing approaches such as salting and slow password hashing algorithms
- examples of stronger choices are `bcrypt`, `scrypt`, or `argon2`

We are not doing that here because the goal of this exercise is to learn authentication and relationships, not production security.

### Mini tutorial: cookie sessions

In this exercise, authentication will use a cookie session.

The idea is:

1. the user logs in
2. the server creates a random session token
3. the server stores that token in the database
4. the server sends that token back in a cookie
5. future requests include that cookie automatically
6. the server reads the cookie and finds the matching user

Useful FastAPI tools for this:

- `request.cookies.get(...)`
- `response.set_cookie(...)`
- `RedirectResponse(...)`

In a real application, cookie sessions should also have an expiration date.

We are not doing that here, because the goal is to learn the main authentication flow first.

### Mini tutorial: what a WebSocket is

A WebSocket is a long-lived connection between the browser and the server.

With normal HTTP:

- the client sends one request
- the server sends one response
- the connection is finished

With a WebSocket:

- the client opens one connection
- the server accepts that connection
- both sides can keep sending messages through that same connection

This is useful for chat because:

- the server can push new messages immediately
- the browser does not need to keep polling again and again

### Mini tutorial: WebSockets in FastAPI

#### A WebSocket route

In FastAPI, a WebSocket route looks like this:

```python
from fastapi import WebSocket


@app.websocket("/ws/chat")
async def websocket_chat(websocket: WebSocket):
    ...
```

#### Accepting the connection

Before using the connection, the server must accept it:

```python
await websocket.accept()
```

#### Receiving data

To receive text:

```python
text = await websocket.receive_text()
```

If you send JSON as text, you can decode it with `json.loads(...)`.

#### Sending data

To send text:

```python
await websocket.send_text("hello")
```

A common pattern is:

- send JSON strings with `json.dumps(...)`
- receive JSON strings with `json.loads(...)`

#### Handling disconnects

When the browser closes the connection, FastAPI raises `WebSocketDisconnect`.

So a common structure is:

```python
from fastapi import WebSocketDisconnect

try:
    while True:
        text = await websocket.receive_text()
        ...
except WebSocketDisconnect:
    ...
```

### Mini tutorial: WebSockets in vanilla JavaScript

#### Open the connection

In the browser, you create a WebSocket with:

```javascript
const socket = new WebSocket("ws://127.0.0.1:8000/ws/chat");
```

In this project, it is better to build the URL from the current page:

```javascript

```

#### Listen for connection events

You can react when the socket opens:

```javascript
socket.addEventListener("open", () => {
  console.log("connected");
});
```

You can react when a message arrives:


You can react when the socket closes:

```javascript

  console.log("disconnected");

```

#### Send a message
